027    /**
028     * A <code>Simplex</code> is the basic unit for simplicial homology
029     * calculations, literally. That is, the module in which simplicial
030     * homology is defined is the free module over all simplices, so a simplex
031     * is actually a basis element for this module. Formally, simplices are
032     * elements of a set S of non-empty finite subsets of a set X which have
033     * the property that if B is an element of S, and A is a non-empty subset
034     * of B, then A is an element of S. Intuitively, if you imagine that the
035     * points of X are in some n-dimensional Euclidean space, then a simplex is
036     * that subset of Euclidean space that is given by the affine combinations
037     * of its "corners". For instance, if B is the triangle given by corners
038     * {a, b, c} is one of our simplices, then we require that the edge (line
039     * segment) given by {a, c} is also a simplex. While we can describe this
040     * relationship in a purely combinatorial fashion, the "meaning" is given
041     * by the choice of simplices S, and this methodology used here always
042     * makes some use of geometric techniques (e.g., metric spaces).
043     *
044     * <p> For reasons of efficiency, we use more than one encoding for
045     * simplices, instead of using the obvious "array of vertices". The reason
046     * for this is that these encodings of simplices can be much smaller, and
047     * the code to manage them much faster, than if we use the obvious
048     * implementation. We make <code>Simplex</code> an abstract class, instead
049     * of an interface, both because code written to this class will be
050     * slightly faster than if it were an interface, and because the root class
051     * is a convenient repository for some utilities needed by the extension
052     * classes, and for "global information" such as the characteristic of the
053     * base field.
054     *
055     * <p> All but the most naive encoding must have, either implicitly or
056     * explicitly, limits on the allowed dimension and number of landmarks that
057     * can be used to construct simplices. By default all limits will either be
058     * the explicit in the representation of the simplex or will be implicitly
059     * the largest values that are feasible for that representation. See the
060     * specific implementations for more details on this point.
061     *
062     * <p> The vertex set of any simplex depends on the context -- that is, for
063     * a Simplex s, s.vertices() is an array of integers of limited size, and
064     * the meaning of those integers will depend on the source of the Simplex
065     * s.  The data sets we operate on frequently have millions of points (and
066     * we hope to push these even further in the future), but we only construct
067     * simplices from much smaller sets, often representative subsets we call
068     * "landmark sets". Since we are able to restrict landmarks to be in the
069     * thousands, or even hundreds, we use a smaller number of bits to encode
070     * the vertex sets for a simplex, often by "indirecting" through an array
071     * of landmark indices to some much larger data set. Thus, a vertex of 10
072     * encoded within an instance of <code>Simplex</code> generally refers to
073     * the point whose index is the 10th entry in some landmark set, not the
074     * 10th point in the full data set.
075     *
076     * @author Harlan Sexton
077     * @version 0.1
078     *
079     */
080    abstract public class Simplex implements Comparable<Simplex> {

083       * The filtration index of the Simplex.  <p> Theoretically this could
084       * be abstract, but in all of the implementations we use this value is
085       * hard enough to find that it makes sense to cache it in the

087       * final without forcing the implementations to compute it during

089       * (see {@link Persistence.java} and references therein) by storing the
090       * reduced boundary chain directly in the Simplex instance. This is
091       * simpler, faster, and may even be smaller (it depends on what
092       * percentage of slots in T[] actually got filled).

224      /**
225       * Overrides Object equals. 
226       *
227       * <p>
228       * This test is equivalent to an equality test on the vertices.
229       * <p>
230       * @param      obj   object to compare.
231       * @return     true or false, depending on whether or not the Simplex is = to obj.
232       *
233       */
234      public boolean equals(Object obj) {
235        if (!(obj instanceof Simplex))
236          return false;
237        Simplex s = (Simplex) obj;
238        int this_dimension = this.dimension();
239        int s_dimension = s.dimension();
240        if (s_dimension == this_dimension)
241          return (0 == Plex.comparePtArrays(this.vertices(), s.vertices()));
242        else 
243          return false;
244      }

246      /**
247       * Implements Comparable interface.
248       *
249       * <p>
250       * This test is equivalent to lexicographic comparison of the vertex arrays.
251       *<p>
252       * @param      s   Simplex to compare.
253       * @return     negative, 0, or positive, if this <, =, resp. > than s.
254       *
255       */
256      public int compareTo(Simplex s) {
257        int this_dimension = this.dimension();
258        int s_dimension = s.dimension();
259        if (s_dimension != this_dimension)
260          return (this_dimension - s_dimension);
261        else 
262          return Plex.comparePtArrays(this.vertices(), s.vertices());
263      }

265      // The seq() and slt() methods are used to perform Chain and
266      // Persistence operations. For instance, simplices in a Chain are
267      // sorted so that arithmetic on Chains is linear in the size of the
268      // instances. The findex() of a simplex is an essential part of the
269      // ordering in the Persistence algorithm, but we don't regard it as
270      // intrinsic to the Simplex otherwise. (The presence of an findex slot
271      // in Simplex is for caching purposes, not because it is essential to
272      // the definition of the type.) Therefore, compareTo(), hash() and
273      // equals() ignore the value of the findex, and changing the findex
274      // slot of a Simplex instance won't change its location in a
275      // hashtable. In particular, it means that we can use a hashtable to
276      // record the findex values of instances of Simplex, if we wish
277      // (although I have no plans to do that at the moment except in initial
278      // test code). To be explicit, the contract for slt() is that it sort
279      // first on findex, second on dimension, and then after that,
280      // lexicographically on vertices. The contract for a.seq(b) is that it
281      // be functionally equivalent to !(a.slt(b) || b.slt(a)) in the places
282      // that it is used, but as an optimization, in the places where slt()
283      // and seq() are called, we can suppose that if the vertices for a and
284      // b are the same, then their findex() values are the same, too.

409       * Add a vertex to a simplex.
410       *
411       * <p> I can't recall why this method exists, so the implementation is
412       * the easiest one I can think of.
413       * @param      v   A vertex to adjoin.
414       * @return     The new simplex.
415       */
416      public Simplex addVertex(int v) {
417        int[] v_old = vertices();
418        int[] v_new = new int[v_old.length+1];
419        for (int i = 0; i < v_old.length; i++)
420          v_new[i] = v_old[i];
421        v_new[v_old.length] = v;
422        return Simplex.getSimplex(v_new);
423      }

440      /**
441       * Represent a Simplex as a String.
442       *
443       * <p>
444       * @return     A string that describes a Simplex.
445       */
446      public String toString() { 
447        int[] vertices = this.vertices();
448        String base = "<";
449        if (_findex >= 0)
450          base = String.format("<(%d)", _findex);
451        if (vertices.length == 0)
452          return base + ">";
453        else if (_findex >= 0)
454          base = base + " ";
455        for (int i = 0; i < vertices.length-1; i++) {
456          base = base + String.format("%d, ", vertices[i]);
457        }
458        base = base + String.format("%d>", vertices[vertices.length-1]);
459        return base;
460      }

462      /**
463       * Sort a set of vertices into increasing order, as needed for some
464       * Simplex operations.  We use the bubblesort algorithm, since that is
465       * fastest for very short arrays, which is what we have in practice.
466       *
467       * <p>
468       * @param      vertices   The vertex set.
469       * @return     The sorted vertices.
470       */
471      public static final int[] vertex_sort (int[] vertices) {
472    
473        if ((vertices.length <= 1))
474          return vertices;
475    
476        for (int j = vertices.length - 1; j > 0; j--) {
477          for (int i = 0; i < j; i++) {
478            if (vertices[i+1] < vertices[i]) {
479              int dummy = vertices[i];
480              vertices[i] = vertices[i+1];
481              vertices[i+1] = dummy;
482            }
483          }
484        }
485        return vertices;
486      }

540      /**
541       * Sort, in place and using "persistence order", an array of
542       * simplices. Null entries will be pushed to the end of the array -- that
543       * is, null is considered to be larger than any non-null Simplex.
544       */
545      public static void persistence_sort(Simplex[] sarray) {
546        PlexSort.comp_sort((Object[])sarray, 0, sarray.length, 
547                           new PlexSort.CompObj() {
548                             public int fn(Object o1, Object o2) { 
549                               if (o1 == o2)
550                                 return 0;
551                               if (o1 == null)
552                                 return 1;
553                               if (o2 == null)
554                                 return -1;
555                               Simplex s1 = (Simplex)o1;
556                               Simplex s2 = (Simplex)o2;
557                               if (s1.slt(s2))
558                                 return -1;
559                               else if (s2.slt(s1))
560                                 return 1;
561                               else
562                                 return 0;
563                             }
564                           } );
565      }

593        if (length > 8)
594          length = 8;
595        if (length < 1)
596          length = 1;
597        int[] vertices = new int[length];
598        for(int i = 0; i < length; i++) {
599          int x = 1 + (int) Math.floor((Math.random() * 1000.0));
600          vertices[i] = x;
601        }
602        vertex_sort(vertices);
603        for(int i = 0; i < length-1; i++) {
604          if (vertices[i] == vertices[i+1]) {
605            for(int j = i+1; j < length; j++) {
606              vertices[j]++;
607    
608            }
609          }
610        }
611          
612        return getSimplexPresorted(vertices);
613      }
